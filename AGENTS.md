# Repository Guidelines

## Project Structure & Module Organization
This repository is a Next.js 14 App Router project.
- `app/`: route segments (`kebab-case`) with `page.tsx` and optional `layout.tsx`.
- `components/`: reusable UI and page-level modules (for example `components/landing-page`, `components/portfolio`, `components/shared`).
- `content/blog/`: markdown posts (many are generated by the daily AI news workflow).
- `public/`: static assets and data files (for example `public/data/portfolio-sample.csv`).
- `lib/`, `utils/`, `types/`: shared helpers and type definitions.
- `scripts/fetch-ai-news.mjs`: RSS-to-markdown ingestion script used by CI.

Use the TypeScript path alias `@/*` for internal imports.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start local dev server at `http://localhost:3000`.
- `npm run lint`: run Next.js ESLint checks.
- `npm run build`: create a production build.
- `npm run start`: run the production server locally.
- `node scripts/fetch-ai-news.mjs`: fetch and generate blog posts from RSS.

Before opening a PR, run `npm run lint && npm run build`.

## Coding Style & Naming Conventions
- Language: TypeScript/TSX with strict mode enabled (`tsconfig.json`).
- Follow existing file-local style (most TS/TSX uses 2-space indentation and no semicolons).
- Route folders use `kebab-case`; app entries use Next.js conventions (`page.tsx`, `layout.tsx`).
- Component filenames are usually `kebab-case`; React component symbols use `PascalCase`.
- Prefer alias imports like `@/components/...` over deep relative paths.

## Testing Guidelines
There is currently no dedicated unit/integration test suite in this repo.
- Treat `npm run lint` and `npm run build` as required validation checks.
- For UI changes, manually verify affected routes in `npm run dev` and include notes in the PR.
- If adding tests, place them near feature code (for example `*.test.ts[x]`) and document the run command in `package.json`.

## Commit & Pull Request Guidelines
Git history favors concise, imperative commits with Conventional Commit prefixes where possible (`feat:`, `fix:`, `chore:`).
- Keep subject lines short and scoped (example: `feat: add geo greeting to homepage`).
- Use `[skip ci]` only for automated content-update commits when appropriate.
- PRs should include: what changed, why, impacted routes/files, and screenshots for UI updates.
- Link related issues/tasks and call out any follow-up work explicitly.
